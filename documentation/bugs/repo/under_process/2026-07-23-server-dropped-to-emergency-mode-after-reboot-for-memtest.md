# Server dropped to emergency mode after reboot for memtest

Status: **mostly resolved** - `tank` is `ONLINE`/healthy, the production site is confirmed reachable, both physical root causes (loose dock cable, BIOS boot-priority conflict) are fixed. Still open: the Memtest86+ pass (deferred, Joakim off-site for ~1 week), the memtest-gate-vs-auto-restart gap, and the linked [smartmontools/udisks2 item](2026-07-23-smartmontools-and-udisks2-services-failed-to-start.md). Not moved to `fixed/` until those close. Keep this file as the full chronological trail as more is learned - don't overwrite conclusions.

## Symptom

Joakim rebooted `192.168.1.10` (`sudo reboot`) intending to hit the boot-menu key (`F12`) and select USB media to run the pending Memtest86+ pass (see [HARDWARE.md](../../../photo-server/HARDWARE.md)'s open memtest gate). The `F12` menu came up as a "choose first boot device" prompt rather than the boot-order-already-USB-first behavior he expected ("that is how I am taught"). He did **not** select USB or CD — chose the normal HDD entry. The machine then dropped into systemd emergency mode instead of booting normally. No USB/CD boot was attempted at any point this incident.

## Investigation log

1. 2026-07-23 — Incident opened. No diagnostic commands run yet; Joakim is at the physical console (screen+keyboard already connected for the memtest work) but the machine is in emergency mode, not a normal shell — SSH is almost certainly down while in this state (network stack not brought up this early in boot).
2. 2026-07-23 — Blocked: root password unknown (never set/recorded — not in HARDWARE.md, this repo has no record of it), so `sulogin`'s emergency-mode password prompt can't be satisfied. Recovery path handed to Joakim: hard power-cycle back to GRUB (unavoidable — no way out of `sulogin` without the password), edit the boot entry (`e`) to append `rw init=/bin/bash` to the `linux` line, boot that one-time edit (`Ctrl+X`/`F10`) to land in a root shell that bypasses systemd/sulogin entirely, no password needed. From there: `mount -o remount,rw /`, then `journalctl -b -1 -xb` (previous boot's log, since this rescue boot itself never ran systemd) plus `cat /etc/fstab`, `lsblk`, `blkid`. Not yet run — awaiting output.
3. 2026-07-23 — **Correction to entry 2**: a hard power-cycle was not actually necessary. The standard `sulogin` emergency-mode prompt reads `Give root password for maintenance (or press Control-D to continue):` — Ctrl+D tells systemd to proceed past the maintenance shell rather than requiring the hard reset the previous entry recommended first. Joakim recalls seeing this prompt but not its exact wording or whether Ctrl+D was pressed yet. Open question for next update: was Ctrl+D pressed, and did it (a) boot through to normal/SSH-reachable, or (b) loop back into emergency mode again — the latter would mean the failed mount/unit isn't skippable and the `init=/bin/bash` route (or a hard-reset-free equivalent, if one hasn't already happened) is still needed for diagnostics.
4. 2026-07-23 — Boot log (before halting) showed `dev-disk-by\x2duuid-C07....device` (a specific partition UUID starting `C07...`) — systemd waiting on a device that isn't showing up, consistent with the loose-cable theory. **Reboot #1 then got all the way to the `ubuntu2404lts` login prompt** (fully successful boot). **Reboot #2, with no configuration change in between, failed differently**: `default.target` failed to load ("default.target is not inactive... fallback to single-user shell" per Joakim's paraphrase, exact wording not yet confirmed). Two different failure modes across two consecutive, unmodified reboots is the more significant finding here — a permanently disconnected cable or wrong fstab entry would fail identically every time; this instead looks like an **intermittent hardware fault**, which raises rather than lowers the priority of actually running the still-pending Memtest86+ pass. Diagnostics requested (`systemctl status default.target`, `systemctl list-units --failed`, `journalctl -xb`, `dmesg`, `fstab`/`lsblk`/`blkid`) — not yet received.
5. 2026-07-23 — **SSH access recovered** (root password still unknown/unset, but Joakim's own `sudo` access is enough). A subsequent boot (this one fully succeeded, hence SSH being reachable at all) shows the real picture in `journalctl -xb`: **all 6 SATA ports on the onboard AHCI controller (`ata3`–`ata8`) report `SATA link down (SStatus 0 SControl 300)`** — no drives detected on any of them. `ata10` (a port on the second, add-in AHCI controller) is also link-down. Only `ata9`, on that same add-in controller, has anything attached — the small 250GB boot drive (`ST3250318AS`). Consequently: `zpool[693]: cannot import 'tank': no such pool or dataset` (the pool's member disk(s) are among the missing drives), and the earlier `dev-disk-by-uuid-C0761D90761D8874` timeout is an `/etc/fstab` entry for a filesystem on one of those same missing drives. `/tank` is currently **not mounted**. This boot did not otherwise fail — whatever unit needed that missing device isn't hard-blocking, hence login/SSH still came up. Read-only follow-up requested (`lsblk`, `blkid`, `sudo zpool import`) to confirm before any physical inspection — not yet received. Also requested: `sudo passwd root`, run interactively at the terminal (value never to be typed into this chat or written to any file) so future incidents don't hit the same root-password wall.
6. 2026-07-23 — Confirmed: `lsblk`/`blkid` show **only** `sda` (the boot drive) system-wide - no other physical block device present at all. `sudo zpool import` (no argument, lists importable pools without importing) returns **"no pools available to import"** - not even `tank` listed as `UNAVAIL`, meaning ZFS finds no pool label on anything, consistent with the pool's drive(s) being completely absent from the OS's view. This is very unlikely to be data loss/corruption - ZFS can't have written anything wrong to disks it can't see; the data should be intact once physical connectivity is restored. Recommended to Joakim: `sudo poweroff` (clean shutdown, safer than probing a live machine), then physically re-check the SATA data cable and power cable/splitter for whichever drive(s) make up `/tank`, specifically on the onboard controller's ports.
7. 2026-07-23 — Root cause of the SATA link-down found: the pool lives in an external 4-bay HDD dock with two possible host connections (USB and SATA, via a breakout cable to the onboard controller) - the SATA cable was unplugged at the **dock end** (still connected at the motherboard end). Reconnected SATA; left the dock's USB cable deliberately disconnected (web-search-confirmed native SATA is more reliable than USB bridge chips for ZFS - see CHANGELOG). This produced a **new** symptom: boot got past the motherboard splash screen but hung on a blinking cursor, top-left, nothing further. Diagnosed (BIOS boot-key confirmed via web search: `DEL` for this board's Award BIOS) as a **boot-priority conflict**: `Advanced BIOS Features > Hard Disk Boot Priority` listed only 3 entries - `Bootable Add-In Cards`, `Ch 4 M: ST3250318AS` (the actual boot drive, on the add-in SATA card), `SCSI-0: P5-WDC WD20EARX-00PASB` (one of the dock's WD drives) - with `Bootable Add-In Cards` ranked *above* the boot drive's own entry. Reordered so `Ch 4 M: ST3250318AS` is first. Reboot succeeded ("seems to work, so far" - Joakim's words). Only 3 boot-priority entries for what should be 5 physical drives (1 boot + 4 dock) is consistent with the still-unresolved "only 2 of 4 dock LEDs lit" observation from earlier in this incident - not yet explained.
8. 2026-07-23 — **Full diagnostic block run before pausing, and the outcome is good.** `lsblk` now shows all 5 physical drives (`sda`-`sdd` = the dock's 4 bays, `sde` = boot drive, device letters shifted since the dock drives sort first). `blkid` clarifies the dock's actual composition: **3 drives are ZFS pool members** (`sda1`/`sdb1`/`sdc1`, `TYPE="zfs_member"`, same pool UUID), and **the 4th (`sdd1`) is a separate NTFS volume** (`LABEL="New Volume"`, mounted at `/media/master_copy`) - not part of the ZFS pool at all. That NTFS drive's UUID (`C0761D90761D8874`) is exactly the device that was timing out at the very start of this incident (entry 4/5) - mystery solved, it was never a ZFS drive. `sudo zpool status`: **`tank` is `ONLINE`, 0 read/write/checksum errors, "No known data errors"** (only a benign block-size-mismatch performance note, not a health issue). `df -h` matches HARDWARE.md's documented `/tank` figures exactly (3.6T, 892G used, 2.7T available), and `/media/master_copy` is also mounted (1.9T, 1.4T used). This resolves the "only 2 of 4 LEDs" and "only 3 boot-priority entries" open questions from entry 7: all 4 dock drives are in fact present and working, the boot-priority list just doesn't enumerate the same drives that `lsblk`/`blkid` do (it showed only 1 WD drive by name, not all 4) - not a real anomaly once the drive composition is understood.
9. 2026-07-23 — **`docker ps -a` shows the full production stack already running**: `photo-viewer`, `auth`, `redis`, `caddy`, `postgres`, all `Up 26-28 minutes` (matching this boot's ~25min uptime) despite their `CREATED` timestamps being 43 hours/2/5 days old - i.e. these are *pre-existing* containers that Docker's own restart policy brought back up on daemon start, not a fresh `docker compose up` that anyone ran. **Confirmed independently**: `curl https://photos.reuterborg.se` returns `200` and serves the real gallery HTML. **This means the memtest gate has been silently bypassed on every reboot during this entire incident** - the gate's documented mechanism ("don't run `docker compose up`") only prevents a *manual* start; it does nothing against `docker-compose.prod.yml`'s own restart policy bringing the stack back automatically whenever the host reboots, which has happened repeatedly tonight. Real gap between the gate's wording and what it actually accomplishes - flagged in HARDWARE.md's gate text, not silently fixed by changing the restart policy itself (that's a separate, deliberate call, not made tonight).
10. 2026-07-23 — Two new, minor findings, split into their own tracked file per this repo's one-bug-per-file convention: [2026-07-23-smartmontools-and-udisks2-services-failed-to-start.md](2026-07-23-smartmontools-and-udisks2-services-failed-to-start.md). Neither blocks the pool or the site. Also benign/unrelated, no action needed: a `gpio_ich` resource-claim message and a `nouveau` (GPU driver) channel error, both pre-existing and irrelevant given this server is treated as CPU-only per HARDWARE.md.
11. 2026-07-23 — **Session closed here.** Joakim is away from physical access to this server for about a week starting today - the still-outstanding Memtest86+ pass is explicitly deferred, not cleared, and can't be scheduled until he's back on-site.

## Root causes (confirmed)

1. **ZFS pool drives going invisible**: the dock's SATA breakout cable was unplugged at the **dock end** (mechanism for how still unconfirmed - plausibly disturbed during the RAM case-opening, but not proven). Fixed by physically reconnecting it; the dock's USB cable was deliberately left disconnected (native SATA is more reliable for ZFS than a USB bridge chip - web-search-confirmed, see CHANGELOG). **Verified fixed**: `tank` is `ONLINE`, all 3 pool members healthy, no data errors.
2. **Boot hang after reconnecting**: this board's Award BIOS `Hard Disk Boot Priority` list gets disrupted when the dock's drive count changes, ranking a generic `Bootable Add-In Cards` entry above the actual boot drive (`Ch 4 M: ST3250318AS`). Fixed by manually reordering. **Standing quirk, not a one-time fix** - likely to recur on any future case-opening or dock reconnection; worth remembering next time rather than re-diagnosing from scratch.
3. **Gate-vs-reality gap** (process finding, not a hardware fault): the memtest gate's "don't run `docker compose up`" wording doesn't actually stop the production stack from restarting on every reboot, because `docker-compose.prod.yml`'s restart policy brings it back automatically regardless. The gate has been effectively bypassed on every reboot tonight. Not fixed this session (would need a deliberate decision on `docker compose down` before any planned reboot, or reworking how the gate is enforced) - flagged in HARDWARE.md.

The new RAM is still not ruled in or out as a contributing factor to any of the above - the memtest pass that would settle that is the one thing still outstanding.
