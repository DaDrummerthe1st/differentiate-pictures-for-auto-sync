# Documentation cleaning pass

A full, deliberately-invoked audit of everything under `documentation/` (plus root `README.md`/`CLAUDE.md`) — distinct from the lightweight, per-session "Doc-drift check" and "Cross-reference link check" rows in [README.md](README.md)'s wrap-up checklist, which are scoped to files a session happened to touch that day. This is a full-repo pass, invoked on demand, not automatically per session — that's deliberate: it doesn't belong in `CLAUDE.md` because it isn't something every session needs to run.

## Goals

Asked for 2026-07-19, in Joakim's own framing: reduce how long a fresh session takes to get up and running, and save tokens doing it — without losing needed context. The premise: the more thorough and precise the audit, the better-placed and better-indexed the resulting information ends up, and *that* is what actually cuts ramp-up time and tokens — not just making files shorter. A doc that's wrong or misleading costs a fresh session far more (in corrected turns) than a doc that's merely verbose costs in tokens.

## What it checks, in order

Order matters here, not just coverage — steps 3-5 are a deliberate sequence, not an unordered checklist: run the scripted, unambiguous checks first (3), then look for a cross-reference that could replace a duplicate before touching prose (4), and only then compact whatever text must stay in place regardless (5). Reversed, this wastes work: independently trimming two duplicate passages before checking whether one of them should just become a link back to the other means re-doing the trim once the link is added anyway.

1. **Read every doc in full**, not just grep — `documentation/**`, root `README.md`/`CLAUDE.md`. Skimming misses contradictions between two sections of the same file.
2. **Cross-check claims against actual code/config**, not just against other docs. A doc can be internally consistent and still be wrong about what the code does.
3. **Scripted checks, run first** (fix everything these flag before moving on to judgment calls):
   - **Dead-link sweep** — every relative markdown link across the repo resolves to a real file. `python3 tools/documentation_checks/run.py` (see [DOCUMENTATION_CHECKS.md](DOCUMENTATION_CHECKS.md) — built 2026-07-19 after this same script got rewritten from scratch, ad hoc, twice in one session). Also covers topic-folder `TODO.md` presence and code-dir stub-README size.
   - **Redundant-phrase scan** — surfaces markdown phrases repeated verbatim across 2+ files, as candidates for step 4/5 below. `python3 tools/redundancy_scan/run.py` (see [REDUNDANCY_SCAN.md](REDUNDANCY_SCAN.md) — built 2026-07-20 after the first manual pass at this was done ad hoc). Every match is a candidate, not an automatic cut — see that doc for real precedent on matches that were reviewed and deliberately kept.
4. **Cross-reference before redundancy**: for every duplicate/near-duplicate passage found (by the scan above or by reading), check first whether it should become a "see X" cross-reference link to a single canonical copy, rather than independently shortened in place. A link is preferred whenever one side doesn't need to stand alone for a reader landing on it directly (per `CLAUDE.md`'s "nothing project-wide duplicated outside `POLICY.md`" rule) — only fall through to compacting both copies in place when each genuinely needs to be self-contained (e.g. a stub `README.md` that must make sense with zero other context loaded, or a `bugs/claude/` incident report quoting the exact rule it violated).
5. **Compact what must stay in place**: for prose that can't just become a link, actively look for adjacent sentences that can merge, connective filler that can go, or two clauses saying the same thing that can collapse into one — not only verbatim-duplicate blocks the scan can find. This is a judgment call a script can't make; the goal from this file's own premise (see Goals above) is precision, not brevity for its own sake — never cut a `**Why:**`/reasoning clause just to shorten a sentence.
6. **`CLAUDE.md` specifically should shrink, not grow, from a cleaning pass.** It's the one file guaranteed to load into every session's context, so accumulated detail there is the most expensive place for it to live — push anything beyond a short rule + pointer out to the relevant topic doc instead (see its own "Documentation stays current" and "Lean, exact, and compact" rules). Check this concretely, not by impression: `grep '"file_path": "CLAUDE.md"' tools/doc_metrics/metrics.jsonl | tail -5` shows its last few logged character counts — the post-pass number should be ≤ the pre-pass one. Growth is acceptable only with an explicit reason in the changelog entry (e.g. a genuinely new rule with nowhere else to live yet), never silent creep.
7. **Structural convention compliance**, per `CLAUDE.md`'s "Documentation layout" section: every "see X" cross-reference terminates in real content within 1-2 hops, not another pointer back; root `README.md` stays the short pitch + pointer, not branch-specific content. (Duplication-outside-`POLICY.md` and code-dir-stub/topic-folder-`TODO.md` presence are covered by steps 3-4 above; cross-reference *termination* specifically still needs a real read — a script can confirm a link resolves, not that what it resolves to is itself just another pointer.)
8. **Fix vs. flag**: fix stale facts and structural violations directly. Flag, don't unilaterally resolve, anything that would mean editing an already-published, append-only record (e.g. a past `CHANGELOG.md` entry) — that needs Joakim's call, not a silent correction.
9. **Verify before committing**: re-run both scripted checks (step 3), run the full test suite, confirm step 6's `CLAUDE.md` size check, log `doc_metrics`/`commit_cost`, commit with a changelog entry detailed enough to serve as the audit's own record.

## Precedent

First run: 2026-07-19, `CHANGELOG.md`'s `(6)` entry that day and the several timestamped entries immediately after it (a changelog-heading convention fix, a `CLAUDE.md` accumulated-detail trim, and a test-rule scoping correction all surfaced *during* that pass, not planned going in — expect a cleaning pass to turn up follow-on fixes beyond the original checklist). Read that entry for what a full pass actually found in this repo, as a concrete example of scope and depth.

Practical note from that run: reading and cross-checking the whole `documentation/` tree against code is large enough to be worth spreading across a few parallel research passes by area (e.g. bugs+tooling, architecture/photo-server, policy+cross-links) rather than one linear read — keeps the pass thorough without exhausting one session's context budget before the fixes even start.

**Protocol formalized 2026-07-20**, at Joakim's request, after the 2026-07-20T19:43:47+00:00 pass (`CHANGELOG.md`) had already done steps 4-5 above manually with an ad hoc phrase scan: the explicit step ordering, the `tools/redundancy_scan/` script, and the `CLAUDE.md`-shrink check (step 6) all came out of that request, so future passes don't have to rediscover the same ordering or rebuild the same scan from scratch.
