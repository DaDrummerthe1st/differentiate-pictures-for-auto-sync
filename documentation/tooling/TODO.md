# tooling/ — open work

- **Replace CHANGELOG.md with one-file-per-entry, mirroring `bugs/`.** Decided 2026-07-23, not yet built. Root cause: merging `master`/`mamma-photo-viewer` that day produced a silently-broken `CHANGELOG.md` (entries out of order, header paragraph buried mid-file) — see `documentation/bugs/claude-bugs/under_process/2026-07-23-changelog-header-paragraph-silently-displaced-by-a-naive-top-insert.md`. A single growing file with multi-line prose entries is exactly the shape git's line-based merge handles worst; web-researched precedent (Towncrier, Changesets — both use one small fragment file per change specifically to avoid this) confirms the fix, but neither tool itself is a good fit here (Changesets needs a Node/npm toolchain this repo deliberately has none of; Towncrier's model assumes versioned releases, which this continuously-deployed repo doesn't have) — adopt the *pattern*, not the tool. Decided shape, all confirmed with Joakim:
  - New `documentation/changelog/` directory, one `.md` file per entry, named `<ISO-timestamp-no-colons>-<slug>.md` (need sub-day precision — multiple entries per day are common; day-only precision like `bugs/` uses won't sort correctly). No index file (same reasoning `bugs/` already documents: tried once, drifted out of sync, removed).
  - A single entry file can span multiple parts of the system (GUI + backend in one file) — the file boundary is "one revision," not "one subsystem." Confirmed against real precedent: Changesets' interactive CLI lets one changeset target multiple packages; Towncrier merges fragments referencing multiple issues into one entry.
  - Existing `CHANGELOG.md` (~161KB, 100 entries) gets frozen as an archive (rename to e.g. `CHANGELOG_ARCHIVE.md`), not retroactively split — avoids rewriting settled history, consistent with the file's own "never rewrite or reorder past entries" rule.
  - Add a helper script mirroring `tools/create_bug_report/create_bug_report.sh` (e.g. `tools/create_changelog_entry/create_changelog_entry.sh "Short title"`) to generate the correctly-timestamped/-slugged file.
  - Still needs doing: the actual directory/README/script; a new row for `changelog/` in `documentation/README.md`'s top-level folder-index table; updating CLAUDE.md's changelog rule to point at the new convention; and a decision on whether/how to render a single browsable view (not required to start — `bugs/` proves `ls`-and-read works fine without one). Already checked, no action needed: `tools/doc_metrics` discovers files generically via `git ls-files -- "*.md"` (`tools/doc_metrics/metrics.py:35`), not a hardcoded list — new `documentation/changelog/*.md` files get picked up with no code change.
